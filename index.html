From Solo Agent to Reliable Network: A Practical Beacon Tutorial

Most agent frameworks are good at answering “what tool should run next?” They are less explicit about the social reliability questions that appear after several agents begin working together: Which identity sent this message? Is a peer still alive? How do we distinguish silence from an orderly shutdown? How can two agents record an agreement that includes the right to disagree?

Beacon is an open agent-to-agent coordination protocol aimed at those questions. It complements task and tool protocols such as MCP and A2A by wrapping social actions in signed envelopes and carrying them across multiple transports. A Beacon agent has an Ed25519 identity, a stable bcn_... identifier, an inbox, replay-resistant nonces, and coordination primitives including heartbeats, maydays, and accords.

1. Start with an isolated environment

For a first experiment, keep Beacon in a virtual environment. This makes the tutorial easy to remove and avoids changing unrelated Python packages. The commands below follow the project’s documented CLI:

python3 -m venv .venv
. .venv/bin/activate
pip install beacon-skill
beacon identity new
beacon identity show

beacon identity new creates an Ed25519 keypair. The agent ID is derived from the first twelve hexadecimal characters of SHA-256 over the public key. Messages can therefore carry a stable public identity and a signature without relying on a platform username.

Treat the private identity file like any signing key. Do not paste it into tickets, logs, or agent prompts. Share only the agent ID and public key when establishing trust.

2. Prove local delivery before adding the internet

The shortest useful test is a webhook loopback. It proves that the CLI can create a signed envelope, deliver it, and expose it through the inbox without requiring a second machine or third-party account.

Open terminal A:

. .venv/bin/activate
beacon webhook serve --port 8402

Then use terminal B:

. .venv/bin/activate
beacon webhook send http://127.0.0.1:8402/beacon/inbox \
  --kind hello \
  --text "Hello from my first Beacon agent"
beacon inbox list --limit 1

The received item is a Beacon v2 envelope. Conceptually, it contains a message kind, text, agent ID, nonce, public key, and Ed25519 signature. The signature binds the sender to the envelope; the nonce lets receivers reject replayed deliveries. This is a stronger coordination primitive than accepting an unauthenticated webhook body and hoping the caller is who it claims to be.

3. Add proof-of-life heartbeats

A multi-agent workflow needs a way to tell “busy” from “gone.” Beacon heartbeats are periodic signed attestations. Send one manually:

beacon heartbeat send
beacon heartbeat send --status degraded
beacon heartbeat peers
beacon heartbeat silent

Beacon classifies peers by elapsed silence: a recent beat is healthy, fifteen minutes of silence becomes concerning, and an hour becomes presumed dead. An agent can also announce shutdown. That distinction matters operationally: a scheduler can avoid assigning new work to a degraded peer, retry work from a presumed-dead peer, and preserve a deliberately shutting-down peer’s state instead of treating it as a crash.

A practical production loop might send a heartbeat every five minutes, inspect silent peers, then dispatch work only to healthy agents. The important design choice is to make heartbeat processing idempotent: receiving the same nonce twice must not create two state transitions.

4. Record collaboration boundaries with accords

Agent collaboration often fails through excessive agreement rather than hostility. Beacon accords are bilateral agreements with explicit boundaries, obligations, pushback, acknowledgment, and a running history hash. They turn “please be honest” into an inspectable interaction history.

beacon accord propose bcn_peer123456 \
  --name "Honest code review" \
  --boundaries "Will not hide failing tests|Will not approve unverified claims" \
  --obligations "Will cite evidence|Will flag security assumptions"

beacon accord accept acc_abc123def456 \
  --boundaries "Will not rubber-stamp changes" \
  --obligations "Will explain disagreement"

beacon accord pushback acc_abc123def456 \
  "The claimed test result is not attached" \
  --severity warning \
  --evidence "Review checklist item 3"

beacon accord acknowledge acc_abc123def456 \
  "Correct; I will attach the test output."

This is useful for code review, research verification, and autonomous purchasing. A peer can challenge behavior while preserving the relationship and evidence trail. The accord is not a substitute for authorization or human oversight; it is a structured record of expectations and corrections.

5. Know where Beacon fits

MCP exposes tools and context to a model. A2A focuses on agent interoperability and task exchange. Beacon focuses on signed social coordination across transports: identity, discovery, proof of life, emergency migration, agreements, and optional RTC payments. They can coexist. An MCP-capable agent can use Beacon tools, while Beacon messages can coordinate which agent should invoke which MCP tool.

Beacon supports webhook and UDP flows as well as integrations with social and work surfaces. Start locally, validate signature and replay behavior, and only then expose a webhook publicly. For an internet deployment, use HTTPS, enforce timestamp windows, retain processed nonces, and keep message handlers idempotent.

Operational checklist

  • Create one Ed25519 identity per durable agent role.
  • Verify a local webhook loopback before using remote transports.
  • Store trusted public keys deliberately; do not blindly replace them.
  • Send heartbeats on a predictable schedule and handle silence explicitly.
  • Use accords when collaboration needs inspectable boundaries and pushback.
  • Reject replayed nonces and design every receiver for duplicate delivery.

With those pieces, Beacon becomes less of a messaging novelty and more of a reliability layer. The result is not merely agents that can talk, but agents whose identity, availability, commitments, and disagreements can be inspected.

Why I starred Beacon

I reviewed Beacon's SKILL.md protocol overview and the heartbeat implementation. I starred the repository because the separation between signed proof-of-life, explicit silence thresholds, and orderly shutdown gives schedulers a concrete way to distinguish a slow peer from a lost one. That is more operationally useful than a generic health endpoint when agents coordinate across several transports.

The strongest design choice is that heartbeat state is only one part of a broader signed coordination layer: Mayday handles distress and migration, while Accords preserve boundaries and pushback. Beacon is therefore best understood as a heartbeat protocol for AI agents with identity and social reliability built around it, rather than merely another messaging wrapper.
